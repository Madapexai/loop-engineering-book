# 第15章 LangChain的四层循环架构

### Agent→验证→事件驱动→爬山

LangChain作为AI代理开发最主流的框架之一，它的架构演进在很大程度上反映了整个行业对Loop Engineering的理解深度。

从最初的Chain（链式调用），到后来的Agent（代理），再到现在的多层循环架构——LangChain的演进本身就是一部Loop Engineering的微观史。

### 15.1 从Chain到Loop的演进

LangChain最早的核心概念是Chain——把多个LLM调用串起来，形成一个处理链。输入→处理1→处理2→…→输出。

这是典型的线性思维——和传统的软件流水线没什么本质区别。

但很快，大家发现线性的Chain不够用了。因为AI代理的工作方式不是线性的——它是循环的。它会试错、会反思、会调整方向、会重新尝试。

于是LangChain引入了Agent的概念——代理可以自己决定下一步做什么、用什么工具、要不要重试。

但Agent只是第一步。单个Agent的循环只是最内层的循环。一个完整的系统，需要多层循环嵌套。

这就是LangChain四层循环架构的由来。

### 15.2 第一层：Agent循环（Agent Loop）

最内层的循环，也是最基础的循环——单个代理的"思考-行动"循环。

#### 工作原理

Agent Loop的经典模式是ReAct（Reasoning + Acting）：

1. **思考（Thought）**：代理分析当前状态，决定下一步做什么
2. **行动（Action）**：代理执行行动（调用工具、写代码、查询信息等）
3. **观察（Observation）**：代理观察行动的结果
4. **回到思考**：根据观察结果，继续思考下一步

这个循环一直进行，直到代理认为任务完成了。

#### 关键特征

- **自主决策**：代理自己决定下一步做什么，不是预设好的步骤
- **工具使用**：代理可以调用各种工具来获取信息或执行操作
- **自我修正**：如果行动结果不对，代理可以调整策略重新尝试

#### 局限性

单个Agent Loop的局限性很明显：

- 上下文有限——任务太长就会"失忆"
- 视角单一——一个代理的思维盲区，它自己很难发现
- 质量不可控——代理觉得做完了，但实际可能做得很糟

这就是为什么需要更外层的循环。

### 15.3 第二层：验证循环（Verification Loop）

第二层循环是验证循环——在代理完成任务之后，再做一轮验证和修正。

#### 工作原理

验证循环在Agent Loop之外，增加了一个"验证者"的角色：

1. Agent完成初步的工作产出
2. Verifier（验证者）检查产出，发现问题和不足
3. 把问题反馈给Agent，让它修正
4. Agent修正后，Verifier再检查
5. 重复直到Verifier满意

这就是我们在第8章讲过的Maker-Checker模式。

#### 为什么需要验证循环？

因为"自己检查自己的工作"是不可靠的。同一个代理的思维盲区，它自己发现不了。换一个代理（甚至换一个模型）来检查，往往能发现很多问题。

#### 验证的维度

验证循环可以从多个维度验证：

- **正确性验证**：结果对不对？有没有bug？
- **完整性验证**：是不是所有要求都满足了？
- **质量验证**：代码质量好不好？设计合不合理？
- **安全验证**：有没有安全漏洞？

#### 关键挑战

验证循环的关键挑战是——**验证者的能力边界**。如果验证者本身能力不够，它发现不了问题，验证循环就形同虚设。

实践中的做法：

- 用更强的模型做验证者（比如用Claude Opus验证Claude Sonnet写的代码）
- 用多个验证者交叉验证
- 用自动化工具（测试、lint、静态分析）做客观验证

### 15.4 第三层：事件驱动循环（Event-Driven Loop）

第三层循环是事件驱动循环——系统监听外部事件，事件到达时触发相应的处理流程。

#### 工作原理

1. 系统持续监听各种事件源（GitHub、Slack、监控系统、数据库…）
2. 当某个事件发生时，根据事件类型和内容，决定启动哪个Loop
3. Loop执行完毕，产出结果，更新系统状态
4. 继续监听下一个事件

这就是我们在第4章讲的Automations的系统级实现。

#### 关键组件

- **事件总线**：接收和分发事件
- **事件处理器**：定义什么事件触发什么操作
- **任务队列**：缓冲事件，控制系统负载
- **状态管理**：跟踪每个任务的状态和进度

#### 为什么重要？

事件驱动循环让系统从"被动响应"变成"主动运转"。没有这一层，你所有的Agent和验证循环都只是工具——需要人来启动。有了事件驱动循环，系统才能自己"活"起来。

### 15.5 第四层：爬山循环（Hill Climbing Loop）

最外层的循环，也是最有想象力的一层——爬山循环。

#### 什么是爬山算法？

爬山算法是一种优化算法：从一个起点出发，每次向"更高"的方向走一步，直到到达山顶。

把这个思路用到Loop系统里：

- 系统有一个"目标函数"——什么是"更好"的状态？
- 系统持续尝试各种改进
- 每次改进后，用目标函数评估是不是变好了
- 如果变好了，就保留这次改进；如果没变好，就回退
- 持续这个过程，系统会"越爬越高"

#### 在Loop系统中的应用

爬山循环可以用来优化很多东西：

- **优化Prompt**：系统自动尝试不同的prompt变体，选效果最好的
- **优化流程**：系统自动调整循环的步骤和参数，找到最优配置
- **优化Skill**：系统自动从成功和失败的案例中学习，更新Skill库
- **优化策略**：系统自动调整重试策略、路由策略、资源分配策略

#### 关键挑战

爬山循环的挑战主要有两个：

**第一，目标函数怎么定义？**  
你怎么量化"更好"？代码质量怎么量化？用户满意度怎么量化？如果目标函数定义错了，系统会"爬"向错误的方向——越优化越糟。

**第二，局部最优问题。**  
爬山算法容易陷入局部最优——爬到一个小山头就以为到顶了，但远处还有更高的山。怎么跳出局部最优，是一个经典难题。

### 15.6 四层循环的嵌套关系

这四层循环不是并列的，而是**嵌套**的：

```
┌─────────────────────────────────────────┐
│          爬山循环 Hill Climbing          │
│  ┌───────────────────────────────────┐  │
│  │      事件驱动循环 Event-Driven      │  │
│  │  ┌─────────────────────────────┐  │  │
│  │  │     验证循环 Verification     │  │  │
│  │  │  ┌───────────────────────┐  │  │  │
│  │  │  │  Agent循环 Agent Loop │  │  │  │
│  │  │  └───────────────────────┘  │  │  │
│  │  └─────────────────────────────┘  │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

越内层的循环转得越快、周期越短；越外层的循环转得越慢、周期越长。

- Agent循环：秒级到分钟级
- 验证循环：分钟级
- 事件驱动循环：分钟级到小时级
- 爬山循环：天级到周级

每一层循环都有自己的节奏和目标，但它们共同构成了一个完整的、自进化的系统。

> **核心洞察**：Loop Engineering不是"写一个循环"。它是设计多层嵌套的循环系统——内层管执行，外层管质量，更外层管调度，最外层管进化。
