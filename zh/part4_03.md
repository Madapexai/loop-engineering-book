### 1个中枢+4条闭环+7个门禁的实战体系

前面的章节介绍了Loop Engineering的概念、模块、通用框架。但如果你想在自己的团队或项目中真正落地，你需要一个更具体、更可操作、经过实战检验的方法论。

这就是M-LOOP框架。

M-LOOP是我们在多个企业级项目中反复打磨出来的实战体系。它不是理论推导的产物，而是从真实项目的成功和失败中总结出来的。

### 17.1 M-LOOP框架总览

M-LOOP的核心架构可以用一组数字来概括：**1个中枢 + 4条闭环 + 7个门禁**。

```
                    ┌──────────────────┐
                    │   Rex 中枢引擎   │
                    │  Research/Route  │
                    │  Review/Rescue   │
                    │     Record       │
                    └────────┬─────────┘
                             │
        ┌────────────────────┼────────────────────┐
        │                    │                    │
  ┌─────▼─────┐        ┌─────▼─────┐        ┌─────▼─────┐
  │ 任务闭环  │        │ 验证闭环  │        │ 协作闭环  │
  │ Task Loop │        │ Verify    │        │ Collab    │
  └───────────┘        └───────────┘        └───────────┘
                             │
                        ┌────▼─────┐
                        │知识闭环  │
                        │Knowledge│
                        └──────────┘
          ┌───────────────────────────────┐
          │  G0 G1 G2 G3 G4 G5 G6         │
          │  七层质量门禁体系              │
          └───────────────────────────────┘
```

**1个中枢**：Rex中枢引擎——整个系统的大脑，负责调度、决策、救援、记录  
**4条闭环**：任务闭环、验证闭环、协作闭环、知识闭环——系统运转的四个轮子  
**7个门禁**：G0-G6七层质量门禁——系统输出的安全网

让我们逐一展开。

### 17.2 Rex中枢：一个大脑管全局

Rex是M-LOOP框架的中央控制系统。它不是一个执行任务的代理，而是一个**管理代理的代理**——它负责让整个系统高效、可靠、有序地运转。

Rex包含五大核心能力（5R）：

#### Research（研究与路由）

任务来了之后，Rex先做分析——这是什么类型的任务？难度多大？风险多高？应该分配给哪个代理团队？需要走哪些流程？

这就像项目经理接到需求后的第一件事——评估、拆解、分配。

#### Route（调度与路由）

根据任务的类型和优先级，Rex决定：

- 用哪个模型？（Opus还是Sonnet？还是本地小模型？）
- 调哪些代理？（前端代理？后端代理？全栈代理？）
- 走什么流程？（简单流程还是完整的7层门禁？）
- 优先级多高？（立即处理还是排队？）

调度的核心是**成本-质量-速度的平衡**。高价值高风险的任务，用最好的资源、最严的流程；低价值低风险的任务，用最轻的配置、最快的速度。

#### Review（审查与把关）

Rex不直接做执行，但它负责关键节点的审查。比如：

- 代理的方案靠不靠谱？要不要打回去重做？
- 测试通过了就真的没问题了吗？
- 这个任务可以自动交付，还是需要升级到人类？

Review不是每一步都检查——那效率太低了。Rex只在关键决策点介入。

#### Rescue（救援与升级）

代理卡住了、失败了、跑飞了——Rex负责救援。

救援策略包括：

- 自动重试（换个策略重试）
- 换代理（换个模型、换个角色的代理来试试）
- 降级处理（降低任务难度，先做核心部分）
- 升级人类（实在搞不定，找人来）

Rescue的核心原则：**永远不要让任务死在半路上。** 失败不可怕，可怕的是失败了没人知道。

#### Record（记录与沉淀）

Rex负责记录整个系统中发生的一切——谁做了什么、为什么这么做、结果如何、花了多少成本。

这些记录不只是为了审计，更是为了知识闭环——所有的经验和教训，最终都会沉淀到Skill库和记忆系统中。

### 17.3 四条闭环：系统运转的四个轮子

#### 闭环一：任务闭环（Task Loop）

**目标**：高效完成单个任务

**流程**：

```
任务输入 → 任务拆解 → 代理执行 → 结果产出
    ↑                        ↓
    └─────────反馈───────────┘
```

任务闭环是最基础的闭环——把一个任务从输入变成输出。它的核心是**代理的执行能力**和**自我修正能力**。

任务闭环对应LangChain四层循环中的Agent Loop层。

#### 闭环二：验证闭环（Verify Loop）

**目标**：确保产出质量

**流程**：

```
任务产出 → 自动化检查 → AI审查 → （可选）人类审查 → 通过/打回
    ↑                                              ↓
    └──────────────────修正───────────────────────┘
```

验证闭环在任务闭环之外，增加了多层质量检查。它的核心是**多视角交叉验证**——不同的代理、不同的工具、不同的维度，共同保证质量。

验证闭环对应LangChain四层循环中的Verification Loop层。

#### 闭环三：协作闭环（Collaboration Loop）

**目标**：多个代理/多个人高效协作

**流程**：

```
任务分配 → 多代理并行/串行工作 → 结果整合 → 冲突解决 → 最终产出
```

协作闭环解决的是"1+1>2"的问题。多个代理一起工作，不是简单的分工，而是需要协调、沟通、整合。

协作闭环的关键机制：

- **任务拆解**：把大任务拆成小任务，每个小任务边界清晰
- **接口定义**：代理之间的交付物有明确的格式和标准
- **冲突仲裁**：当多个代理的结果有冲突时，有机制解决
- **人类介入点**：复杂的协作决策，由人来拍板

#### 闭环四：知识闭环（Knowledge Loop）

**目标**：系统越用越聪明

**流程**：

```
任务执行 → 结果记录 → 经验提炼 → Skill/记忆更新 → 指导下一次任务
    ↑                                                    ↓
    └────────────────────────────────────────────────────┘
```

知识闭环是最重要也最容易被忽略的闭环。它让系统从"每次从零开始"变成"站在过去的经验之上"。

知识闭环的核心产出：

- **可复用的Skill**：从成功案例中提炼出的最佳实践
- **失败模式库**：从失败案例中总结的常见坑和避坑方法
- **项目记忆**：项目特有的知识、规范、偏好
- **优化建议**：基于数据分析的流程和策略优化建议

> **核心洞察**：很多团队的Loop系统做了很久，但效果提升不明显。为什么？因为他们只有前三个闭环，没有知识闭环。系统一直在跑，但没有在学习。跑得再多，水平也不会提高。

### 17.4 七层质量门禁（G0-G6）

质量门禁是M-LOOP框架的安全网。我们在第11章已经详细介绍过七层门禁，这里做一个简要回顾和在M-LOOP框架中的定位：

| 门禁 | 名称 | 检查内容 | 执行者 |
|-|-|-|-|
| G0 | 语法门禁 | 编译、lint、格式 | 自动化工具 |
| G1 | 测试门禁 | 单元测试、集成测试 | 自动化测试 |
| G2 | 静态分析门禁 | 安全扫描、代码质量 | 静态分析 + AI |
| G3 | AI审查门禁 | 设计、可读性、最佳实践 | Checker代理 |
| G4 | 集成门禁 | 集成测试、回归测试 | CI/CD系统 |
| G5 | 预演门禁 | 预发环境验证 | 自动化 + AI |
| G6 | 人工门禁 | 业务逻辑、设计决策 | 人类工程师 |

在M-LOOP框架中，门禁不是"所有任务都走全套"。Rex中枢会根据任务的类型、风险、优先级，**动态配置门禁的层数和严格程度**。

比如：

- 改个错别字 → 只走G0+G1
- 加个小功能 → 走G0-G3
- 改核心模块 → 走G0-G5
- 改支付/安全相关 → 走G0-G6全套

这种**动态门禁**的设计，既保证了高风险任务的质量，又不会让低风险任务承担不必要的成本和延迟。

### 17.5 配套机制

除了核心的"1+4+7"架构，M-LOOP还有几个配套机制：

#### 飞书@验证铁律

这是一个简单但极其有效的机制：**任何Loop产出的内容，如果涉及到人的协作（发通知、写文档、回复消息），必须有一个"@相关人确认"的步骤。**

为什么叫"铁律"？因为这是不容商量的底线——它保证了人永远在边界上，Loop不会在人不知情的情况下做影响他人的事情。

#### 30分钟主动汇报

任何运行中的Loop，如果30分钟内没有任何进展输出，必须主动向负责人汇报当前状态和遇到的问题。

这个机制防止了"Loop卡了但没人知道"的情况——Loop可以慢，但不能失联。

#### 审计日志

所有操作都有完整的审计日志——谁（哪个代理）、在什么时候、做了什么、为什么、结果如何。

审计日志不只是为了追责，更是为了分析和优化——通过分析日志，你能发现系统的瓶颈、代理的弱点、流程的问题。

#### 外力二验

对于高风险任务，除了系统内部的验证，还会引入"外力"来做二次验证——可能是另一个团队的代理，可能是不同公司的模型，也可能是外部的安全审计服务。

"自己查自己"永远有盲区。外力二验是打破盲区的有效手段。

#### Skill沉淀机制

每个任务完成后，系统会自动分析：这次任务有没有什么新的经验可以沉淀成Skill？有没有踩了什么新坑需要加入失败模式库？

知识闭环不是空喊口号——它有具体的机制来保证。

### 17.6 M-LOOP的落地路径

M-LOOP框架不是让你一步到位——它是渐进式的。你可以按照这个路径逐步落地：

**第一阶段：搭建任务闭环（1-2个月）**

- 先把单个任务的执行流程跑通
- 用最简单的触发方式（手动触发也行）
- 用最基础的质量检查（G0+G1）
- 目标：能自动完成一些简单任务

**第二阶段：加上验证闭环（2-3个月）**

- 增加AI审查门禁（G3）
- 建立Maker-Checker模式
- 开始度量成功率和准确率
- 目标：产出质量稳定，不需要太多人工返工

**第三阶段：建设协作闭环（3-4个月）**

- 支持多代理分工协作
- 建立Rex中枢的调度能力
- 增加Rescue救援机制
- 目标：能处理复杂的、需要多角色协作的任务

**第四阶段：完善知识闭环（持续进行）**

- 建立Skill沉淀机制
- 建设项目记忆系统
- 开始做流程优化和策略迭代
- 目标：系统越用越聪明，持续改进

> **M-LOOP哲学**：不要追求完美，要追求进步。每个阶段都有明确的目标，达成了就前进，没达成就停下来优化。Loop是持续运转的，Loop系统的建设也是持续的。
