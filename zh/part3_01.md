# 第10章 设计你的第一个Loop

### 从零开始的五步方法论

好了，你已经理解了Loop Engineering的核心概念和六大模块。现在，你想自己动手设计一个Loop。从哪里开始？

很多人的第一反应是："我要搭一个完整的系统，六大模块一个都不能少。"

错。

设计Loop的正确姿势，不是从架构开始，而是从**一个具体的痛点**开始。找到一个让你最头疼的、重复性最高的任务，把它变成你的第一个Loop。

下面是经过验证的五步方法论。

### 第一步：选对任务 —— 不是所有任务都适合Loop化

第一步也是最关键的一步：**选对第一个任务。** 选好了，事半功倍；选错了，你可能从此对Loop Engineering失去信心。

什么样的任务适合做第一个Loop？

**好的候选任务有这些特征：**

- ✅ **重复性高**：每周/每天都要做，频率足够高
- ✅ **定义清晰**：输入是什么、输出是什么、成功标准是什么，都很明确
- ✅ **风险可控**：即使做错了，后果也不严重，可以很容易地回滚
- ✅ **有验证方法**：能自动判断结果对不对，不需要人每次都看
- ✅ **单步骤/少步骤**：不要一上来就搞复杂的多代理协作

**坏的候选任务：**

- ❌ 一年才做一次的任务（投资回报太低）
- ❌ "优化用户体验"这种模糊的目标（没法验证）
- ❌ 直接改生产数据库的任务（风险太高）
- ❌ 需要深度业务判断的任务（AI搞不定）

> **经验法则**：你的第一个Loop，应该是那种"做对了没什么大不了，但做错了也不会死人"的任务。先建立信心，再逐步升级。

### 第二步：定义清楚 —— 输入、输出、成功标准

选定任务之后，不要急着写代码。先把三个问题回答清楚：

**1. 输入是什么？**  
Loop从哪里获取任务？是GitHub Issue？是飞书消息？是定时触发？输入的格式是什么？有哪些字段是必需的？

把输入想象成一个函数的参数——参数定义清楚了，函数才能写对。

**2. 输出是什么？**  
Loop做完之后要交付什么？是一个PR？是一段分析报告？是一个评论？是一封邮件？

输出的格式同样要明确。比如PR的标题格式、描述模板、需要打什么标签。

**3. 成功的标准是什么？**  
这是最重要也最容易被忽略的问题。怎么判断Loop做的事情是"对"的？

成功标准可以是：

- 测试通过了
- Lint没有报错
- 代码编译成功
- 满足了某个验收条件
- （对于无法自动验证的任务）被人类批准了

没有清晰的成功标准，Loop就会像第12章讲的那样——变成一个没有退出条件的永动机。

### 第三步：设计流程 —— 画出你的Loop图

定义清楚输入输出之后，就可以设计Loop的内部流程了。

一个最简单的Loop流程是这样的：

```
触发 → 接收任务 → 代理执行 → 验证结果 → 成功？→ 是 → 交付
                                      ↓否
                                    重试？→ 是 → 代理执行（带错误反馈）
                                      ↓否
                                    升级到人类
```

设计流程的时候，问自己几个问题：

- **需要几步？** 尽量简单，第一步能做对就不错了
- **谁来做决策？** 每一步的决策主体是代理还是系统？
- **失败了怎么办？** 重试几次？每次重试有什么不同？什么时候放弃？
- **人什么时候介入？** 完全不需要人？还是最后需要人确认？

建议你在纸上或者白板上画出来——可视化的流程图能帮你发现很多想的时候发现不了的问题。

### 第四步：最小实现 —— 先跑起来，再优化

很多人设计Loop的时候陷入"完美主义陷阱"——要做最优雅的架构、最全的功能、最好的体验。结果做了三个月还没上线。

记住这个原则：**第一个版本能用就行。**

你的第一个Loop可以非常简陋：

- 不用MCP，直接调用CLI工具
- 不用Worktree，就在一个目录里干活（反正任务简单，冲突概率低）
- 不用Skill，把所有说明都写在prompt里
- 不用Memory，每次从零开始
- 不用Sub-agent，就一个代理干到底

简陋没关系，重要的是——**它真的能跑，真的能帮你省时间。**

等它跑起来了，你再根据实际使用中的痛点，逐步优化。哪块最疼，就先优化哪块。

这和做产品是一个道理：先有MVP（最小可行产品），再迭代。

### 第五步：度量和迭代 —— 让Loop越跑越好

Loop上线不是结束，而是开始。

你需要度量一些关键指标，来判断Loop的效果，并指导下一步优化：

- **成功率**：多少任务成功完成了？多少失败了？
- **准确率**：成功的任务里，有多少是真的做对了（不需要返工）？
- **平均耗时**：完成一个任务平均需要多久？
- **人工介入率**：多少任务需要人来处理？
- **成本**：每个任务平均花多少Token/钱？

拿到这些数据之后，你就知道该优化什么了：

- 成功率低？→ 优化prompt、增加验证步骤
- 准确率低？→ 加Checker代理、加强质量门禁
- 人工介入率高？→ 分析介入原因，针对性优化
- 成本太高？→ 优化上下文、减少重试次数、用更小的模型

> **核心观点**：Loop本身也是一个迭代的循环。你设计了Loop，然后Loop的运行数据反过来指导你优化Loop。这才是真正的"循环工程"。

### 案例：一个工程师的第一个Loop

小王是一个后端工程师，他最烦的事情就是每次有人提bug Issue，他都要花时间分类——这个bug属于哪个模块？严重程度如何？是不是重复的？

他决定用Loop来自动化这件事。

**第一步，选任务**：Issue分类。重复性高（每天都有）、定义清晰（输入是Issue，输出是标签）、风险低（标错了改回来就行）、有验证方法（人可以抽查）。

**第二步，定义清楚**：

- 输入：GitHub Issue的标题和描述
- 输出：给Issue打上模块标签和优先级标签
- 成功标准：标签准确率达到80%以上（人抽查验证）

**第三步，设计流程**：

```
新Issue创建 → 触发Loop → 代理读取Issue内容 → 代理判断模块和优先级 → 打标签 → 结束
（失败了就重试一次，还不行就跳过，等人类来处理）
```

**第四步，最小实现**：用GitHub Actions + Claude Code的API，写了几十行脚本。没有Worktree，没有Skill，没有Memory，就是最朴素的实现。

**第五步，迭代**：

- 第一周：成功率60%，经常标错模块。→ 加了项目模块说明文档作为上下文。
- 第二周：成功率80%，但优先级判断不准。→ 加了优先级判断标准的详细说明和示例。
- 第一个月：成功率90%，基本不用管了。→ 加了Memory，把历史分类结果存下来，越用越准。

现在，这个Loop每个月帮小王节省了大约8小时的时间。而他花在搭建和优化上的时间，总共不到10小时。

**这就是Loop Engineering的投资回报：前期花一点时间，后期持续受益。**
